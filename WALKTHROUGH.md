# 변형 스도쿠 플레이어 — 진행 경과 정리

## 0. 시작 상태: 프로젝트 복구

작업을 시작했을 때 프로젝트는 다른 IDE(구 세션)에서 발생한 크래시로 인해 `src/main.js`와
`style.css`가 통째로 사라져 있어 앱이 아예 실행되지 않는 상태였다. 남아있던 단서는:

- 삭제되지 않고 남아있던 나머지 모듈들 (`Board`/`Cell`/`Structure`/`Validator`, `structures/*`,
  `puzzles/*`, `BoardRenderer`/`DragPanel`/`Keypad`) — 문법적으로는 멀쩡했다.
- `scratch/restore_board.js`, `scratch/find_board.js` — 크래시 이전 세션이 다른 IDE의
  트랜스크립트 로그를 뒤져 `BoardRenderer.js`를 복구하려 시도한 흔적. 이 복구는 성공적이었던
  것으로 보인다 (해당 파일이 문법 오류 없이 온전한 상태로 남아있었음).
- `implementation_plan.md`, `prompt.txt` — 애초 설계 의도와 그동안 요청했던 기능/수정 이력.

이 두 문서를 근거로 빠져있던 `src/main.js`(보드·키패드·드래그패널 조립, 이벤트 바인딩)와
`style.css`(디자인 시스템)를 새로 작성해 앱을 다시 동작하게 만들었다. (두 파일은 이후 정리되어
현재는 삭제됨 — 프로젝트가 안정화된 뒤 더 이상 참조할 필요가 없어짐.)

## 1. 초기 복구 직후 발견/수정한 핵심 버그

- **키패드 클릭·드래그가 먹통이 되는 문제**: `board-wrapper`는 CSS `transform: scale()`로
  축소되지만, 부모 요소의 레이아웃 박스 크기는 축소 *전* 원본 크기(십자 보드 전체,
  약 1192×1192px) 그대로 남아 화면 대부분을 덮는 보이지 않는 히트박스가 됐다. 여기에 더해
  `DragPanel`이 클릭할 때마다 z-index를 최상단으로 올리는 로직이 있어, 보드 칸을 한 번이라도
  클릭하면 보드가 키패드보다 위로 올라오고 그 거대한 히트박스가 키패드 클릭을 전부 가로챘다.
  → `BoardRenderer.setScale()`을 만들어 확대/축소 시마다 wrapper의 실제 레이아웃 크기를
  시각적 크기와 동기화하도록 수정, 이후 보드의 `DragPanel`에 `noZBoost: true`를 추가해 보드를
  클릭해도 키패드보다 위로 올라오지 않게 함.
- **키패드 버튼 클릭 시 활성 숫자 하이라이트가 갱신되지 않는 문제**: `BoardRenderer.inputValue()`가
  값 입력 후 `onCellSelect` 콜백을 호출하지 않던 누락 버그. 추가해서 해결.

## 2. 보드 렌더링

- 9×9 판 전체 외곽선을 3×3 박스 경계선보다 두껍게(`GRID_THICK`) 그리도록 `_drawGridBorders()`
  추가, 충돌(빨간 테두리) 표시 두께도 여기에 맞춤.

## 3. 디자인

- 파란 계열 → 눈에 덜 피로한 연한 초록 계열 팔레트로 전환 (`--cell-selected`, `--text-input` 등).
- 보드는 패널 없이 그림자만 있는 형태로, 키패드는 글라스모피즘 패널로 유지.

## 4. 키패드 패널 확장

- **모두 지우기**: 키패드에 "모두 지우기" 버튼 추가 — 클릭 시 확인 모달(배경 클릭/Esc로 취소,
  Enter로 확정)이 뜬 뒤에만 실행되도록 함.
- **추가 메뉴 토글**: 처음엔 토글 버튼이 키패드 자체를 접었다 폈다 하는 방식으로 만들었으나,
  요청에 따라 "토글은 별도의 추가 메뉴 패널을 여닫는 것"으로 재설계. 추가 메뉴는
  `position:absolute`로 flex 흐름 밖에 두고 키패드 왼쪽 가장자리에 폭 0→N으로 펼쳐지게 해서,
  본 키패드 위치가 밀리거나 접혔을 때 안 보이는 히트박스가 남는 문제를 피함.
  이후 "토글 버튼이 추가 메뉴보다 항상 왼쪽에 있어야 한다"는 요청에 맞춰 토글+추가메뉴를
  `kp-drawer`라는 하나의 flex 단위로 묶어, 추가 메뉴가 펼쳐지면 토글도 함께 왼쪽으로
  밀려나도록 재조정.

## 5. 메모(후보 숫자) 기능

- `Cell.candidates` (기존에 있었지만 안 쓰이던 `Set<number>` 필드)를 메모 저장소로 재사용.
- "메모" 버튼(왼쪽 Shift 키로도 토글) — 메모 모드에서 숫자를 누르면 실제 값이 아니라 칸을
  123/456/789 위치로 나눈 작은 숫자로 표시.
- 메모가 있는 칸에 일반 모드로 메모에 없는 숫자를 입력하면 입력은 무시되고 메모 숫자들이
  빨간색으로 짧게 점멸, 메모에 있는 숫자를 입력하면 정상적으로 값이 기록되고 메모는 지워짐.
- "지우기"는 모드에 상관없이 값 + 메모를 항상 함께 지우도록 통일.

## 6. 실행 취소 (Ctrl+Z / Cmd+Z)

- 값 입력·메모 토글·지우기·모두 지우기 각각이 `{행, 열, 이전 값, 이전 메모}` 스냅샷으로
  undo 스택에 쌓이고, Ctrl+Z로 한 단계씩 되돌림. "모두 지우기"는 지워진 모든 칸이 하나의
  undo 단계로 묶여 한 번에 복원됨. 스택 크기 제한은 두지 않음 — 판이 커봐야 몇백 칸이라
  메모리 부담이 거의 없기 때문 (새로고침하면 히스토리는 사라짐, 별도 영속화는 안 함).

## 7. 가장 최근 라운드

- 키패드 조작 버튼을 메모 / 지우기 / 실행 취소(꺾인 화살표 "↩") 3개로 통일해 한 줄에 배치.
- 추가 메뉴를 "맨 아래 도움말 버튼 고정 + 위쪽은 스크롤 가능한 기능 버튼 목록" 구조로 재구성
  (`kp-extra-scroll` / `kp-extra-help`).
- **저장 / 불러오기**: 추가 메뉴에서 열리는, 키패드 패널과 같은 지위의 독립 패널. 3개 슬롯에
  `localStorage` 기반으로 저장(`Board.serialize()`)/불러오기(`Board.loadSerialized()` +
  `BoardRenderer.refresh()`) 가능, 빈 슬롯은 "불러오기" 버튼이 비활성화됨. 오른쪽 위 ✕로 닫기.
- **도움말**: 마찬가지로 독립 패널, 지금까지의 모든 키보드 단축키를 나열. 오른쪽 위 ✕로 닫기.
- 단축키 확장: `H` = 도움말 토글, `Ctrl/Cmd+S` = 저장/불러오기 토글, `Esc` = 열린 저장/도움말
  패널 닫기. 확인 모달이 떠 있을 때나 저장/도움말 패널이 떠 있을 때는 보드 조작 단축키
  (숫자·방향키·WASD·메모·Undo)가 새어 들어가지 않도록 가드 처리.

## 8. 멀티플레이어 — 개요

싱글플레이 전용이던 앱에 실시간 멀티플레이(배틀/협동 모드, 방 생성·참가, 대기실)를 추가하는
요청. 범위가 매우 커서 한 번에 만들지 않고 5단계로 나눠 진행했고, 단계마다 Plan Mode로
설계를 먼저 정리해 승인을 받은 뒤 구현했다.

백엔드는 처음엔 PythonAnywhere(WebSocket 미지원) 배포를 염두에 두고 Flask + 폴링으로
설계했으나, "웹소켓 지원하는 플랫폼을 써도 되니 개발이 원활한 쪽으로 하자"는 방향 전환에 따라
**Node.js(Express + `ws`)**로 완전히 갈아탔다. 프론트가 이미 100% 바닐라 JS라 언어 전환
비용이 없고, 폴링 대신 진짜 push가 가능해진다는 점이 결정적이었다. 상태 저장은 순수 메모리
(`Map`) — 방은 카훗류처럼 원래 일회성이라 서버 재시작 시 날아가도 괜찮다는 판단.

프로토콜은 하이브리드로 정리됐다: 방 생성·참가·설정변경·시작·나가기 같은 **명령은 REST**로
(테스트하기 쉽고 상태 변화가 명확), 참가자 목록/게임 상태 같은 **실시간 동기화는 WebSocket
push**로(`/ws?code=<code>&token=<token>` 접속 시 최초 상태 1회 + 이후 변경마다 자동 push,
클라이언트가 따로 폴링하지 않음). 이 하이브리드 골격은 이후 협동 모드의 셀 입력·턴테이블
회전·커서 동기화까지 그대로 확장해서 썼다.

## 9. Phase 1 — 랜딩 화면

- 앱 진입 시 "싱글플레이 / 멀티플레이" 두 버튼이 뜨는 랜딩 화면 추가. 싱글플레이는 기존
  게임 화면 그대로, 멀티플레이는 같은 카드 안에서 하위 화면(방 생성/참가)으로 전환.
- 키패드 추가 메뉴에 "메인 화면으로" 버튼 추가(확인 모달 포함).
- 게임 화면이 안 보이는 동안 숫자·화살표·WASD·undo 같은 게임 단축키가 새어 들어가지 않도록
  `isGameActive()` 가드 추가.

## 10. Phase 2 — Node.js 백엔드

`server/index.js`(Express 앱 + `http.createServer` 위에 `WebSocketServer` 부착, 레포 루트를
그대로 정적 서빙)와 `server/rooms.js`(순수 로직 모듈 — Express/`ws`에 의존하지 않아 단위
테스트하기 쉬움, 메모리 저장소)로 구성. 방 코드는 4자리 숫자, 참가자 인증은 생성/참가 시
발급되는 `X-Player-Token` 헤더. 호스트가 나가면 가장 먼저 들어온 참가자에게 자동으로 방장이
넘어가고, 아무도 안 남으면 방이 삭제된다.

## 11. Phase 3 — 방 생성·참가 + 대기실 UI

- 랜딩 화면의 "방 생성"/"방 참가"가 실제 폼(닉네임, 모드 토글, 참가자 수, 퍼즐 템플릿)으로
  이어지고 대기실 화면으로 진입. 대기실은 참가자 목록(본인 닉네임만 인라인 편집 가능), 방장
  전용 설정 컨트롤, "게임 시작"(방장만) 전부 WS push로 실시간 갱신.
- CSS `[hidden]` 속성 vs 요소 자체 `display` 규칙의 우선순위 문제를 여기서 처음 만났다 —
  브라우저 기본(`[hidden]{display:none}`)은 UA 계층이라 요소 자신의 author-origin `display`
  규칙(`.landing-stack{display:flex}` 등)에 밀린다. `.landing-stack[hidden]{display:none;}`
  처럼 명시적으로 덮어써야 해결됨. 이후 `.wr-body[hidden]`에서 같은 문제를 한 번 더 만남.

## 12. Phase 4 — 배틀 모드

- 방장이 "게임 시작"을 누르면 로컬에서 `generatePuzzle(template)`을 호출해 만든
  `{structures, givens}`를 서버로 올리고, 서버는 이걸 그대로 다른 참가자들에게 중계(서버는
  퍼즐 생성 로직을 몰라도 됨). 각 클라이언트는 받은 구조체를 `reviveStructures()`로 실제
  클래스 인스턴스로 복원해 각자 독립된 보드를 구성 — 배틀은 "같은 배치를 각자 푸는" 개인전.
- 타이머는 서버가 정한 시각 하나(`playingStartedAt`)만으로 전원이 동일하게 계산해서
  클라이언트 시계 오차 문제를 원천 차단. 시작 시 3초 동기화 카운트다운도 같은 원리.
- 리더보드: 완주자는 완주 순서대로 고정, 항복자는 "항복" 표시. 항복(forfeit)과 나가기(leave)를
  분리한 게 핵심 — 처음엔 항복 즉시 방을 나가게 했더니, 다른 참가자의 최종 결과 화면에서
  항복한 사람이 통째로 사라지는 버그가 났다(항복으로 게임이 끝나는 것과 실제 퇴장이 같은
  타이밍에 겹쳐 두 번째 브로드캐스트가 첫 번째를 덮어씀). 항복해도 참가자 목록엔 남아있게
  고쳐서 해결.
- `Box3x3`/`grid9x9` 구조체가 `originRow`/`originCol`라는 베이스 클래스에 없는 전용 필드를
  들고 있다는 걸 놓쳐서, 구조체를 범용으로 복원하는 첫 버전이 이 필드를 날려먹어 배틀 모드
  보드에 `NaN` 좌표 SVG 에러와 깨진 테두리가 났었다. `reviveStructures()`에 타입별 분기를
  추가해 해결.

## 13. Phase 5 — 협동 모드

싱글/배틀과 가장 다른 지점: 협동은 전원이 **하나의 보드**를 함께 푼다. 그래서 서버가 opaque
중계자가 아니라 **진짜 authoritative한 `Board` 인스턴스를 직접 들고 있는다** — 클라이언트가
쓰는 것과 완전히 같은 `src/core/*`, `src/structures/*` 코드를 서버(Node)에서도 그대로
import해서 쓴다(둘 다 DOM에 의존하지 않는 순수 로직이라 가능했음). 규칙 검증 로직을 서버용으로
따로 구현할 필요가 없어진 게 이 설계의 핵심 이득.

- **값 입력 동기화**: 클라이언트는 값을 로컬로 바로 바꾸지 않고 의도만 서버에 WS로 보낸다.
  서버가 given 칸 여부 등을 검증해 실제로 반영한 뒤 결과를 전원(입력한 사람 포함)에게
  브로드캐스트하면, 그때 모든 클라이언트가 동일한 코드 경로로 화면을 갱신한다. "내 입력"과
  "남의 입력"을 구분하지 않는 단일 경로라 낙관적 반영 후 롤백하는 복잡한 경우의 수가 아예
  없다. 부수 효과로, 로컬 undo 스택에 애초에 아무것도 안 쌓이니 Ctrl+Z가 별도 처리 없이도
  자연스럽게 무해한 no-op이 됨.
- **커서 공유**: 값 입력과 반대로 순수 낙관적 처리 — 커서는 "틀릴 수 있는" 개념이 아니라서
  로컬 선택은 즉시 반영하고 동시에 서버로 보내 다른 참가자에게만 relay. 칸 위에 참가자별 색
  테두리 + 그 아래 닉네임 태그(글자 수에 맞춰 자동으로 크기가 늘어나는 배경 알약)로 표시.
- **"누가 채웠는지" 표시**: 값이 채워진 칸 오른쪽 아래에 채운 사람 색의 작은 삼각형 마커.
  플레이어 색은 참가 순서(`colorIndex`)로 10색 팔레트에서 배정.
- **진행 현황판**: 배틀 리더보드와 같은 자리·모양의 오른쪽 위 패널에, 순위 대신 참가자별
  닉네임 + 실시간 채운 칸 수를 표시.
- **재접속**: 서버에 8초 재접속 유예 시간을 둬서(소켓이 끊겨도 곧바로 방을 나간 것으로
  처리하지 않음) 새로고침해도 튕겨나가지 않게 했고, 클라이언트도 방 code/token을
  `sessionStorage`에 저장해뒀다가 페이지 로드 시 자동으로 재접속을 시도한다. 재접속하면 서버가
  현재 보드 스냅샷(`coopCells`)을 그대로 다시 push해줘서 진행 상황이 복원된다.
- **턴테이블 회전 동기화 버그**: 처음 구현 땐 회전이 협동에서 전혀 공유되지 않았다 — 회전
  커밋 코드가 `structure.rotate(this.board, steps)`를 직접 호출해 로컬 보드만 바꾸고, 값
  입력에 만들어둔 서버 동기화 경로를 아예 타지 않았기 때문(구현할 때 놓친 지점). 값 입력과
  똑같은 패턴(서버가 회전을 적용하고 결과를 브로드캐스트, 클라이언트는 확정 후에만 반영)으로
  고쳤고, `Turntable.rotate()`가 옮기는 값/given 여부/메모에 "누가 채웠는지"도 같이 실어
  옮기도록 확장했다(안 그러면 회전 후 마커가 엉뚱한 칸에 남음).
  - 고치고 나서 "회전한 게 자기 화면에서도 적용이 안 보인다"는 두 번째 신고를 받았는데,
    원인이 두 가지였다. 하나는 실제 버그: 드래그를 놓는 순간 미리보기를 일단 원래 모습으로
    되돌린 뒤에 서버 응답을 기다렸어서, 서버가 확정해주기 전까지 아주 짧게 "회전 전으로
    돌아갔다가 다시 회전하는" 깜빡임이 있었다 — 되돌리는 대신 목표 각도(90°×steps)로 스냅해
    그대로 들고 있다가 서버 확정이 오면 그 시점에 실제 값으로 갈아끼우도록 고쳐서, 자기
    화면에서도 체감상 즉시 반영되게 만들었다. 다른 하나는 진짜 원인: 사용자가 테스트하던
    포트 3000 서버가 이 세션 초반부터 계속 떠 있던 **오래된 프로세스**여서, 정적 파일과
    달리 재시작해야만 반영되는 `server/*.js` 쪽 수정이 전혀 적용되지 않고 있었다. 서버
    프로세스를 재시작하고 나서야 실제로 해결됨 — 새 기능을 서버 쪽에서 고칠 때마다 "재시작
    필요"라는 걸 새삼 확인한 사례.
- **저장/불러오기를 협동에서도 쓸 수 있게**: 저장은 손댈 필요 없었다(로컬 `board`가 이미
  서버와 동기화된 실시간 미러라 그대로 스냅샷 떠도 정확함). 불러오기는 원래 로컬 보드를
  직접 덮어써서 협동의 "공유 보드" 원칙을 깼기 때문에, 새 REST 엔드포인트(`/coop-load`)로
  스냅샷을 서버에 보내 authoritative 보드에 반영한 뒤 기존 `roomState` 브로드캐스트로
  전원에게 퍼지도록 바꿨다. 이 과정에서 "참가자 목록이 바뀌는 것처럼 불러오기와 무관한
  push에도 매번 전체 스냅샷을 다시 적용하게" 만들었더니, 가만히 있는 다른 칸들의 로컬 메모까지
  덩달아 지워지는 부작용이 생겨서 — 실제로 값이 바뀐 칸만 건드리도록 조건을 추가해 고쳤다.
- **멀티플레이 화면 정리**: 퍼즐 선택/자동 생성/타이머 토글 버튼은 방 단위로 퍼즐이 고정되는
  배틀·협동 양쪽에서 아예 의미가 없어서 숨김 처리(기존엔 비활성화만 해서 계속 보였음).
  저장/불러오기는 보이되, 배틀은(개인전 페어니스상) 계속 비활성, 협동은 활성화.

## 검증 방식

매 라운드마다 로컬 정적 서버(`python -m http.server`, 멀티플레이 이후엔 `node server/index.js`)
+ Playwright 헤드리스 브라우저로 실제 클릭/드래그/키보드 입력을 재현해 콘솔 에러 유무와
스크린샷으로 눈으로 직접 확인한 뒤에만 완료로 보고했다. 특히 z-index/히트박스 관련 버그들은
코드만 봐서는 잘 드러나지 않아서, 실제 클릭 좌표에서 `document.elementFromPoint()`로 어떤
요소가 클릭을 받는지 직접 찍어보는 방식으로 근본 원인을 추적했다.

멀티플레이어부터는 여러 브라우저 컨텍스트(플레이어 2~3명)를 동시에 띄워 실시간 동기화를
검증하는 방식을 썼다 — 한쪽 화면에서 클릭·드래그·키 입력을 재현하고, 폴링 없이 다른 화면들에
결과가 push로 반영되는지 확인. `page.on('websocket')`으로 실제 WS 프레임(요청/응답 페이로드)을
가로채 서버가 정확히 뭘 보냈는지 직접 확인하기도 했고, 회전처럼 DOM 순서가 바뀔 수 있는
상호작용은 배열 인덱스가 아니라 셀의 실제 (x,y) 좌표로 매칭해야 테스트 스크립트 자체가
틀리지 않는다는 것도 이 과정에서 확인했다.
